# SMS — AWS Infrastructure (Terraform)

Provisions the production cloud for the School Management System per the CLAUDE.md
target: **ECS Fargate + Terraform + GitHub OIDC + CloudFront + WAF + ALB + private
subnets + NAT**, with RDS Postgres (RLS), ElastiCache Redis, S3 Document Vault and
Secrets Manager.

## Architecture

```
            Route53 (A/alias)
                  │
        CloudFront (us-east-1 ACM)  ──  WAFv2 (CLOUDFRONT scope, us-east-1)
                  │  https-only, injects X-Origin-Verify secret
                  ▼
   ALB (public subnets, regional ACM, TLS1.3)
                  │  443, rejects requests without the secret header (403)
                  ▼
   ECS web service (Next.js :3000, private subnets)
                  │  http://api.sms.local:3001  (Cloud Map discovery)
                  ▼
   ECS api service (NestJS :3001, private subnets)
            ├── RDS Postgres 16  (data subnets, no internet route)  ← RLS
            ├── ElastiCache Redis (data subnets)
            ├── S3 documents bucket (CMK-KMS, versioned, private; presigned URLs)
            └── Secrets Manager (auth/crypto/db urls/paystack)

   EventBridge (daily) ──> one-off retention task (migrate creds, isolated)
                            purges expired integrity telemetry across tenants
```

The **Document Vault** uses a real S3 presigner (`STORAGE_PROVIDER=s3`): the API
issues short-lived presigned PUT/GET URLs signed with the task role, so bytes move
browser↔S3 directly. Objects are SSE-KMS with a customer-managed key (`kms.tf`);
the API task role is scoped to that key + bucket only.

Three subnet tiers per AZ (x2 AZ): **public** (ALB, NAT), **private** (ECS tasks,
egress via NAT), **data** (RDS, Redis — *no* default route, isolated). Postgres and
Redis are never reachable from the internet.

## Security posture (maps to the Golden Rules)

- **Two DB roles (Rule #4).** The RDS master user is the *privileged migration*
  role (`sms_migrator`). The least-privilege app role (`major_user`) is created by
  the one-off migrate task, not by RDS, and is what the API connects as so RLS is
  enforced.
- **No secrets in code (Rule #6).** Every credential is generated by Terraform
  (`random_password`/`random_id`) into Secrets Manager and injected into containers
  via ECS `secrets`. The Terraform **state therefore contains secrets — keep the S3
  state bucket private + encrypted**, and rotate after bootstrap.
- **Defense in depth at the edge.** WAF managed rules + rate limit on CloudFront;
  the ALB only accepts requests carrying the CloudFront `X-Origin-Verify` secret, so
  it cannot be reached directly bypassing WAF.
- **Least-privilege IAM.** ECS execution role reads only the app secrets; the API
  task role gets scoped S3 access to the documents bucket only; the GitHub deploy
  role can push to these ECR repos and update these ECS services, nothing else.
- **TLS end to end.** CloudFront→ALB is `https-only`; the ALB redirects :80→:443.
- **Privileged work is isolated, not standing.** Schema migrations and the NDPR
  integrity-telemetry purge run as the table-owner role on **short-lived scheduled
  tasks** (the migrate task on deploy; an EventBridge-driven retention task daily).
  The internet-facing API service only ever holds the least-privilege `major_user`
  credentials, so it can never `DELETE` integrity telemetry or alter schema.

## Prerequisites

- A Route53 **hosted zone** for your domain (set `route53_zone_id` + `domain_name`).
- Terraform ≥ 1.6, AWS credentials with admin for the first apply.
- Docker images get pushed by CI; the first apply creates empty ECR repos and the
  ECS services will sit at 0 healthy tasks until the first `deploy.yml` run.

## Apply order

```bash
cp terraform.tfvars.example terraform.tfvars   # fill in domain/zone/github_repo
terraform init                                  # local state for the first apply
terraform apply
```

(Optional, recommended) move state to S3 afterwards:

```bash
# create the bucket + lock table once (see backend.tf.example header), then:
cp backend.tf.example backend.tf                # edit the bucket name
terraform init -migrate-state
```

## First deploy (and every release)

CI (`.github/workflows/deploy.yml`) does it automatically on push to `main`:

1. Assume the OIDC deploy role (set repo secret `AWS_DEPLOY_ROLE_ARN` to the
   `github_deploy_role_arn` output).
2. Build + push `api` and `web` images to ECR, tagged with the git SHA.
3. **Run the one-off `sms-prod-migrate` ECS task and wait for exit 0.** It
   bootstraps `major_user`, runs `prisma migrate deploy`, applies the ordered RLS
   SQL, and seeds — all as the privileged migrate role.
4. Register the new task defs and `update-service`; wait for services-stable.

To run the migrate task by hand:

```bash
aws ecs run-task --cluster sms-prod --task-definition sms-prod-migrate \
  --launch-type FARGATE --network-configuration \
  "awsvpcConfiguration={subnets=[<private-subnet-ids>],securityGroups=[<api-sg>],assignPublicIp=DISABLED}"
```

## Notes / TODO before going live

- `paystack_secret_key` defaults to empty (Paystack disabled / graceful 503). Set it
  in `terraform.tfvars` or rotate the `paystack-secret-key` secret directly.
- Redis in-transit TLS is gated by `redis_transit_encryption` (default `false` for
  the t4g.micro dev tier). Set it `true` for prod: Terraform generates an auth token
  into the `redis-auth-token` secret and the API connects with TLS + `REDIS_PASSWORD`
  automatically. (At-rest encryption is always on.)
- `deletion_protection` is on for RDS and the ALB. `terraform destroy` needs it
  cleared first — intentional, to avoid accidental data loss.
- The local Docker Compose stack (`infrastructure/`) is unchanged and still the dev
  path; this directory is the cloud target only.
