# Platform-owner org model

> How the super_admin platform owner is modeled (a system org, not a customer school)

*Engineering note (project) — a point-in-time record from a build session. The durable spec is [CLAUDE.md](../../CLAUDE.md); verify details against the code before relying on them.*

---

The super_admin is the **platform owner** (sells the SMS to schools), NOT a member of any customer school. Because `user`/`audit_log` are tenant-scoped with non-null `school_id` + RLS, a truly null-school user would break the model. So the super_admin lives in a dedicated **platform organization**: a `school` row with `isPlatform=true` (slug `sms-platform`, name "SMS Platform"), seeded in `packages/db/prisma/seed.ts` alongside `owner@sms.platform`. The owner's `User.schoolId` and its super_admin `UserRole.schoolId` both point at the platform org — NOT the `demo` school (they used to point at demo; that was the inconsistency).

`School.isPlatform` (migration `20260712000000_school_is_platform`; `school` is the global RLS-exempt registry so no RLS file). The platform org is excluded from every customer surface: `PublicService.listSchools` (`isPlatform: false`), `OperatorService.listTenants` (`where isPlatform:false`), and `DirectorySearchService.run` (`and.push({ school: { isPlatform: false } })`). It has no `SchoolSubscription`, so billing/dunning already skip it.

Operator cross-tenant audits (subscription set, impersonate, student view, onboarding review, analytics view) are written under the operator's OWN tenant (now the platform org) in a SEPARATE `runAsTenant(this.ctx(p))` tx — never inside the target-school tx, or the audit_log RLS WITH CHECK rejects it (the `operator-subscription-audit-rls` bug). The affected school is kept in `metadata.targetSchoolId`. These operator audits are now BEST-EFFORT (try/catch + logger warn) via `OperatorService.auditAsOperator` / `OperatorProvisioningService.auditInOperatorTenant` / `PlatformAnalyticsService.auditView` — because `audit_log.schoolId` has an FK to `school`, a STALE super_admin session (school_id pointing to a platform-org id that no longer exists after the dev DB was recreated) would otherwise 500 the dashboard + onboarding-approve AFTER the privileged write already committed. Real-world fix for a stale session: log out + back in (the platform org is stable in prod). The action is also captured by the observability request log.

Onboarding (`POST /operator/tenants`) lets the super_admin choose a plan (BASIC/STANDARD/ENTERPRISE) + extra force-on modules via `overrides.enabled` (same model as the subscription PUT); the web `Provisioning.tsx` has a plan select + add-on-module checkboxes (mirrors `SubscriptionManager.tsx`).

**Platform-owner business analytics**: `PlatformAnalyticsService` (`apps/api/src/operator/platform-analytics.service.ts`) reads cross-tenant via the shared `PrivilegedDatabaseService` (RLS-bypassing). Excludes the platform org everywhere. Endpoint `GET /operator/analytics` (`platform.operate`, audited `operator.analytics.view` under the operator's own tenant); 503 when no privileged URL. Rendered by `PlatformAnalytics.tsx` on the super_admin **/dashboard** (NOT /operator anymore).

**EXTENDED (2026-07-01) — decision-grade SaaS metrics** in `PlatformAnalyticsDto` (`packages/types/src/dto/operator.ts`): `mrr` (monthly recurring revenue = Σ perSeatMonthly(effectivePlan)×seats for ACTIVE subs; seats fall back to that school's student count; +byPlan, arpaMinor, payingSchools), `growth` (last-6-months series {month,schools,students,revenueMinor} from School.createdAt / student user.createdAt / PAID payments), `funnel` (requests→approved→provisioned→paying), `risk` (pastDue/canceled counts + atRiskMrrMinor from PAST_DUE subs), `moduleAdoption` (schools-per-module via resolveModules over each school's effective plan+overrides), `topSchools` (by students, with plan+MRR), `averages`. Unit test `platform-analytics.service.spec.ts` updated (mock needs `school.createdAt`, `user.findMany`, `userRole.schoolId`, `subscription.seats/overrides`, `studentProfile.findMany`).

**Cross-tenant audit console (2026-07-01)**: `PlatformAuditService` (`apps/api/src/operator/platform-audit.service.ts`, privileged client) reads `audit_log` across ALL customer schools (platform org excluded) and RESOLVES each actor to email + `uniqueId` + roles (+ school name) — so any change/approval by a school's principal/school_admin is fully attributable. Filters: schoolId, actorEmail (contains), role (narrows to userIds holding that role), action (contains), entity, from/to, limit(≤2000). `GET /operator/audit` (list) + `GET /operator/audit/export.csv` (StreamableFile CSV report, RFC-4180 escaped) — both `platform.operate`, both meta-audited (`operator.audit.view`/`operator.audit.export`). DTO `PlatformAuditEntryDto`. Web: `/operator/audit` page + `PlatformAudit.tsx` client (filter bar + table showing name/email/uniqueId/role-badges + Export CSV via the binary BFF); nav key `operatoraudit` ("Platform audit", `ScrollTextIcon`) added to `PLATFORM_OWNER_NAV` + platform group. Unit test `platform-audit.service.spec.ts` (5 cases). NOTE: audit_log columns are id/schoolId/actorId/action/entity/entityId/metadata/createdAt (NO updatedAt).

**Student demographics (2026-07-01)**: gender/age/state come from `StudentProfile` (`gender` free-text, `dateOfBirth`, `state`). Pure bucketing helpers `normalizeGender` (→ Male/Female/Other) + `ageBand`/`ageYears` in `@sms/types` (`packages/types/src/demographics.ts`). School `AnalyticsService.overview` adds `demographics {profiled, gender, ageBand, state}` (staff school-wide, gated `student.profile.read` — principal + school_admin both have it); platform `PlatformAnalyticsDto.demographics {profiled, gender, ageBand}` across all customer schools. Charted on `/analytics` (gender donut + age columns + state bars) and the super_admin `/dashboard` (gender donut + age columns). "Each profile parameter chartable" = gender/ageBand/state breakdowns; extensible by adding more `Record<value,count>` maps.

**Nav for the platform owner**: `AppShell.tsx` restricts a `platform.operate` holder to `PLATFORM_OWNER_NAV` (dashboard, operator, directory, notifications, account) — hides tenant-operational pages (Analytics/Games/etc.).

**Dashboard vs Operator (fixed 2026-07-01 — they used to be the SAME page):** `/dashboard` no longer redirects super_admin → `/operator`. Now `/dashboard` = a GRAPHICAL platform overview (renders `PlatformAnalytics` from `/operator/analytics` — KPIs + donut charts for schools-by-plan/status, onboarding-pipeline bars, revenue trend + latest payments), and `/operator` = the management console (provisioning, subscriptions, tenants, impersonation) with the analytics block REMOVED. Each cross-links to the other. Charts use a dependency-free SVG kit `apps/web/components/charts/charts.tsx` (`Donut`/`Bars`/`Columns`/`Kpi`/`CHART` palette — pure, server-renderable, on-brand). Same kit powers the school `/analytics` page. School analytics gained a **grade distribution** (`AnalyticsOverviewDto.grades` — A/B/C/D/F bands + averagePct, gated `grade.read`, staff school-wide + family-scoped); `AnalyticsService.overview` computes it from PUBLISHED grades. Seed fix: `school_admin` was MISSING `grade.read` in `seed.ts` (had it in the `GRADEBOOK_ROLE_PERMISSIONS` constant but not the seed) — added `grade.read`+`grade.write`; `principal`/`board` already had read. So school_admin + principal now see graphical attendance/grade/fees analytics for their school.
